1、准备说明

最近计划出一期介绍mq相关内容，以rabbitmq为例，关于安装和环境很早之前已经搭建完成，当时没有做笔记，在linux环境下安装rabbitmq的部分这次就不体现了，有兴趣的朋友可以自行查找资源安装。

2、前期准备

本期内容主要针对对rabbitmq有一定基础的同学，如果0基础可以自己补补课。
网上有一张关于各种mq之间的对比关系，在今后技术选型上可能有些帮助

RabbitMQ的五种队列模式示例，本期主要是通配符模式Topic
简单模式Hello World：一个生产者P发送消息到队列Q,一个消费者C接收
工作队列模式Work Queue：一个生产者，多个消费者，每个消费者获取到的消息唯一，多个消费者只有一个队列
发布/订阅模式 Publish/Subscribe：开始引入交换机的概念，一个生产者发送的消息会被多个消费者获取。一个生产者、一个交换机、多个队列、多个消费者
路由模式Routing：生产者发送消息到交换机并且要指定路由key，消费者将队列绑定到交换机时需要指定路由key
通配符模式Topic：生产者P发送消息到交换机X，type=topic，交换机根据绑定队列的routing key的值进行通配符匹配；

  • 符号#：匹配一个或者多个词 lazy.# 可以匹配 lazy.irs或者lazy.irs.cor
  • 符号*：只能匹配一个词 lazy.* 可以匹配 lazy.irs或者lazy.cor

4、rabbitmq的探究

（1.1）、关于投递的可靠性(生产者)

rabbitmq对消息的可靠性提供了两种解决方案，

  • confirm 确认模式
  • return 回退模式

rabbitmq消息投递的路径：
producer –> rabbitmq –> broker –> exchange –> queue –> consumer
confirm确认模式是在producer发送消息的时候，可以在发送端设置一个confirmCallback回调实现监听，将来是否到达交换机都会触发回调进行告知。
return回退模式是在交换机到queue之间，如果exchange到queue传递失败了，就会返回一个returnCallback，而且只有在失败的时候会触发。

实现confirm确认模式的方式，需要在yml修改配置文件添加一个publisher-confirms: true

配置完成之后确认的回调方法就会被执行QueueMessageServiceImpl.java

参数1：correlationData：相关配置信息，这个参数是在发送消息的时候进行设定的，

实现的方式，将自己定义的接口中定义出这个方法，通过引用RabbitTemplate.convertAndSend 的方式去实现

（4）、死信队列（dlx）

死信队列主要针对与两个及以上的队列之间的交互，当一个生产队列出现问题时，或者消息生产出来时候没有及时的被消费，此时可以在过期之间将消息发送给绑定交换机，由绑定交换机的队列进行后续处理的一种模式。

消息在以下三种情况会成为死信：

  • 队列消息长度达到限制
  • 消费者拒签 basicNack，并且不报消息放回原来的队列
  • 原队列的消息过期
    实现的方式：设置参数 x-dead-letter-exchange 和 x-dead-letter-routing-key
    生产端配置两套交换机和队列，其中一个是正常队列，一个是备用死信队列，正常队列需要通过 x-dead-letter-exchange 和 x-dead-letter-routing-key关联到死信队列 调用队列观察发现队列的数据移动
    Queue item_queue -> Queue item_queue_dead

（5）、延迟队列

通过使用TTL消息过期时间的设置实现延迟队列的效果， TTL+死信队列
当一笔数据因为一些原因产生的延迟，导致消息没有被消费者消费，此时消息就会淤堵在mq中，这就好比一个延迟的效果，为了防止消息队列的淤积影响正常队列的进行，此时可以将队列的数据设定过期的时间，过期的对别直接进入到死信队列中，同时设定消费队列监控这个死信队列，一旦死信队列的有消息进来，消费队列就会启动消费，将队列中的数据获取出来，结合业务实际的情况通过查询数据库等校验，判断数据是否达到了理想的效果，如果没有达到预期的效果，要将数据回滚会之前的状态，这里最需要注意的是，消费队列主要监控的是死信队列的routingId，主要的代码实现可以参照以上TTL和死信队列部分。

5、应用

（1）、消息的可靠性

主要采用消息的补偿机制
Produce 生产者将数据保存到业务数据库，操作成功之后发送消息给队列，Consumer接收到消息并进行消费，同时操作自己的数据库，
如果Produce操作数据库成功了，但是消息发送的时候失败了，这样一来消费端也会对应的失败，此时可以采取以下方案：
Produce发送消息完成之后，延迟在发送一条消息，这两条消息一样但是不是一个队列，正常消息称为Q1，补充消息称为Q2，此时消费端要对消息进行消费，消费完成要发送一条消息给一个新的队列Q3，说明自己消费完成，Q3会将数据写入到数据库表中。延迟消息Q2到达时，此时需要一个服务区数据库中比对当前的数据是否在消费端数据库中，如果存在，就说明该消息已经处理过了，补偿消息作废，如果不存在说明消息没有被消费，这样就通知生产端，重新在发送消息，或者将当前的消息入库，这样就可以保障数据的一致性。

（2）、消息的幂等性

幂等性保障不是mq独有的说法，一次或者多次请求资源的时候，每次请求的资源都是一样的。mq采用乐观锁机制保障幂等性
生产者在发送消息的时候每条消息都设定一个version版本号，Consumer对应消费，如果生产方传输多条消息并且version相同的话此时就会引发重复消费。
为了解决这个问题可以通过数据库的方式确保一致性，第一条数据执行是 会触发数据库更新操作update table set version = version+1 where version= 1 and id = id,此时第二条数据到达时，就不会查询到满足条件的数据，update更新就会失败，这样就会确保数据的幂等性。
